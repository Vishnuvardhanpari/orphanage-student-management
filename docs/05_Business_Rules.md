# Business Rules

## Overview

This document defines the business rules governing the Orphanage Management System (OMS). These rules ensure consistency, data integrity, security, and maintainability across the application.

---

# Student Lifecycle

A student progresses through the following lifecycle:

```
New Admission
      ↓
Student Registration
      ↓
Active Student
      ↓
Student Information Updates
      ↓
Student Leaves Orphanage
      ↓
Soft Deleted (Inactive)
      ↓
Historical Record Retained
```

A student record must never be physically deleted from the database.

---

# Student Registration Rules

* Every student must have a unique Admission Number.
* Admission Number cannot be modified after creation.
* Student Name is mandatory.
* Date of Birth is mandatory.
* Gender is mandatory.
* Admission Date is mandatory.
* Student status defaults to **ACTIVE**.
* A student photo is recommended but optional.
* Aadhaar Number is optional but must be unique if provided.

---

# Student Update Rules

Users may update:

* Personal Information
* Guardian Information
* Educational Information
* Medical Information
* Student Photo
* Supporting Documents

Every update must update audit fields.

---

# Student Exit Rules

When a student leaves:

* Record Exit Date.
* Record Exit Reason.
* Change Status to INACTIVE.
* Perform Soft Delete.
* Preserve all historical information.

Student records must remain available for reporting.

---

# Soft Delete Rules

Soft Delete must:

* Set deleted = true
* Record deleted_by
* Record deleted_date
* Change status to INACTIVE

The record must remain in PostgreSQL.

Default searches must exclude deleted students.

Administrators may search historical records.

---

# Restore Rules

Only Administrators may restore students.

Restore shall:

* deleted = false
* deleted_by = null
* deleted_date = null
* status = ACTIVE

---

# Student Documents

Student documents are managed as part of the Student Profile.

There is no standalone Document module.

Documents may only exist if a Student exists.

Every uploaded document belongs to exactly one student.

---

# Allowed Document Types

* Student Photograph
* Aadhaar Card
* Birth Certificate
* Medical Certificate
* Mark Sheet
* Transfer Certificate
* Identity Proof
* Other Supporting Documents

---

# File Upload Rules

Allowed formats:

* PDF
* JPG
* JPEG
* PNG

Maximum upload size:

10 MB

Rejected uploads:

* Executable files
* Scripts
* Unsupported formats

---

# File Storage Rules

Actual files shall be stored in Google Cloud Storage.

PostgreSQL stores only:

* File Name
* File Type
* Storage Path
* Uploaded By
* Uploaded Date
* File Size

---

# Search Rules

Search should support:

* Partial Name
* Admission Number
* Aadhaar
* Guardian Name
* Mobile Number

Search should be case insensitive.

---

# Filter Rules

Users may filter by:

* Status
* Gender
* Admission Year
* Leaving Year
* School
* Age
* Created Date

Filters may be combined.

---

# Report Rules

Reports may be generated for:

* One Student
* Multiple Selected Students
* Filtered Students

Generated PDFs should include:

* Student Photo
* Student Information
* Uploaded Document References
* Generated Date
* Generated By

---

# Security Rules

Every request except authentication must require JWT authentication.

Only authorized roles may access secured endpoints.

Passwords shall be encrypted using BCrypt.

Google OAuth2 users shall also be assigned application roles.

---

# Audit Rules

The following actions shall be audited:

* Login
* Logout
* Student Created
* Student Updated
* Student Deleted
* Student Restored
* Document Uploaded
* Document Replaced
* Report Generated

Audit fields:

* created_by
* created_date
* updated_by
* updated_date
* deleted_by
* deleted_date

---

# User Management Rules

* Users must be pre-provisioned by an Administrator. Open self-registration is not supported.
* Soft disable only — user records are never physically deleted.
* Disabling a user persists `enabled = false` and revokes all active refresh tokens.
* Enabling a user sets `enabled = true` and clears lockout counters (`failedLoginAttempts`, `lockUntil`, `accountNonLocked`).
* An Administrator cannot disable their own account.
* The last enabled Administrator cannot be disabled or demoted.
* Password reset may upgrade a `GOOGLE` auth provider user to `LOCAL_GOOGLE` and revokes refresh tokens.

---

# Validation Rules

Mandatory fields must be validated on both frontend and backend.

Backend validation is the source of truth.

No client-side validation alone is sufficient.

---

# Performance Rules

All list APIs shall support:

* Pagination
* Sorting
* Filtering

No API shall return unlimited records.

---

# Cost Optimization Rules

The application shall minimize infrastructure costs by using:

* Angular hosted on Google Cloud Storage
* Spring Boot hosted on Cloud Run
* PostgreSQL hosted on Supabase
* Student documents stored in Google Cloud Storage

The application shall avoid unnecessary cloud services.
